# O que são Dados?

## Definição de Dados

Dados são fatos brutos, figuras e estatísticas que por si só não possuem significado contextual. Eles apresentam elementos primários de informação que precisam ser processados para se tornarem úteis.

## Características dos Dados

- **Elementares:** Unidades básicas de informação
- **Não processados:** Carecem de contexto e significado
- **Quantificáveis:** Podem ser medidos e armazenados
- **Neutros:** Não possuem interpretação inerente

## Exemplos de Dados

```text
25.4
"João Silva"
"2024-03-15"
1500.00
```

## Hierarquia da Informação

```text
Dados → Informação → Conhecimento → Sabedoria

Exemplo:
	Dado: 25.5
	Informação: Temperatura de 25.5ºC
	Conhecimento: 25.5ºC é uma temperatura agradável
	Sabedoria: Usar roupas leves nessa temperatura
```

# O que é um Banco de Dados?

![[Pasted image 20250923161005.png]]
## Definição Formal

Um banco de dados é uma coleção organizada de dados estruturados, armazenados eletronicamente em um sistema computacional. É projetado para gerenciar, armazenar e recuperar informações de maneira eficiente.

## Componentes Fundamentais

```sql
-- Estrutura básica de um banco de dados relacional
DATABASE → TABELAS → REGISTROS → CAMPOS
```

## Características Essenciais

- **Persistência:** Dados mantidos permanentemente.
- **Integração:** Dados relacionados são agrupados.
- **Controle de Redundância:** Minimiza duplicação.
- **Consistência:** Manutenção da integridade dos dados.
- **Segurança:** Controle de acesso e permissões.
- **Concorrência** Múltiplos usuários simultaneamente.

# Sistemas de Gerenciamento de Banco de Dados (SGBD)

## O que é um SGBD?

Software que permite criar, manter e manipular bancos de dados. Atua como interface entre o banco de dados e os usuários/aplicações.

## Funções Principais de um SGBD

1. **Definição de Dados:** Criar e modificar estruturas.
2. **Manipulação de Dados:** Inserir, atualizar, consultar.
3. **Controle de Acesso:** Gerenciar segurança.
4. **Backup Recovery:** Proteger contra perdas.
5. **Controle de Concorrência:** Gerenciar acesso simultâneo.

## Exemplos de SGBDs Populares

| **SGBD**        | **Tipo**    | **Linguagem** |
| --------------- | ----------- | ------------- |
| Oracle Database | Comercial   | PL/SQL        |
| MySQL           | Open Source | SQL           |
| PostgreSQL      | Open Source | PL/pgSQL      |
| SQL Server      | Comercial   | T-SQL         |
| SQLite          | Embedded    | SQL           |

# Banco de Dados Relacional

## Conceito Fundamental

Baseado no Modelo Relacional proposto por Edgar F. Codd em 1970, organiza dados
em tabelas (relações) com linhas (tuplas) e colunas (atributos).

## Estrutura de uma Tabela Relacional

```sql
CREATE TABLE Clientes (
	id INT PRIMARY KEY,
	nome VARCHAR(100) NOT NULL,
	email VARCHAR(255) UNIQUE,
	data_cadastro DATE DEFAULT CURRENT_DATE
);
```

## Princípios do Modelo Relacional

1. **Tabelas**: Estruturas bidimensionais
2. **Chaves Primárias**: Identificadores únicos
3. **Chaves Estrangeiras**: Relacionamentos entre tabelas
4. **Integridade Referencial**: Consistência nos relacionamentos

## Exemplo Prático de Relacionamento

```sql
-- Tabela Pai
CREATE TABLE Departamentos (
    id INT PRIMARY KEY,
    nome VARCHAR(100) NOT NULL
);

-- Tabela Filha
CREATE TABLE Funcionarios (
    id INT PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    departamento_id INT,
    FOREIGN KEY (departamento_id) REFERENCES Departamentos(id)
);
```

## Vantagens dos Bancos Relacionais

- **Consistência**: ACID Properties    
- **Flexibilidade**: Consultas complexas
- **Padronização**: Linguagem SQL
- **Segurança**: Controle granular de acesso

# Banco de Dados Estruturado

## Definição

Bancos onde os dados seguem um scheme pré-definido e rígido. Possuem estrutura fixa e organizada.

## Características

- **Schema Fixo:** Estrutura predefinica
- **Dados Tabulares:** Organização em linhas e colunas
- **Tipos de Dados Definidos:** Cada campo tem tipo específco
- **Relacionamentos Explícitos:** Links bem definidos entre dados

## Exemplos de Dados Estruturados

```sql
-- Exemplo de tabela estruturada
CREATE TABLE Produtos (
	codigo INT PRIMARY KEY,
	nome VARCHAR(200) NOT NULL,
	preco DECIMAL(10,2),
	categoria VARCHAR(50),
	estoque INT DEFAULT 0
);
```

## Aplicações Típicas

- Sistemas transacionais (OLTP)
- Aplicações financeiras
- Sistemas de gestão empresarial (ERP)
- Bancos de dados relacionais tradicionais

# Banco de Dados Não Estruturado

## Definição

Armazenam dados sem schema fixo pré-definido. Podem conter diversos formatos e estruturas variáveis.

## Características Principais

- **Schema Dinâmico:** Estrutura flexível.
- **Dados Heterogêneos:** Múltiplos formatos.
- **Escalabilidade Horizontal:** Expansão por adição de nós.
- **Alta Performance:** Otimizado para leituras.

## Tipos de Dados Não Estruturados

```json
// Exemplo de documento JSON (não estruturado)
{
  "usuario": {
    "id": 12345,
    "nome": "Maria Silva",
    "preferencias": {
      "cores_favoritas": ["azul", "verde", "roxo"],
      "notificacoes": {
        "email": true,
        "sms": false,
        "push": true
      }
    },
    "historico_compras": [
      {"data": "2024-01-15", "produto": "Livro", "valor": 45.90},
      {"data": "2024-02-20", "produto": "Mouse", "valor": 89.90}
    ]
  }
}
```

## Exemplos de Bancos Não Estruturados

| Tipo        | Exemplos              | Casos de Uso                 |
| ----------- | --------------------- | ---------------------------- |
| Documentos  | MongoDB, Couchbase    | Catálogos, perfis de usuário |
| Chave-Valor | Redis, DynamoDB       | Cache, sessões de usuário    |
| Grafos      | Neo4j, Amazon Neptune | Redes sociais, recomendações |
| Colunar     | Cassandra, HBase      | Analytics, big data          |

# Comparativo: Estruturado vs Não Estruturado

|Característica|Estruturado|Não Estruturado|
|---|---|---|
|**Schema**|Fixo e rígido|Dinâmico e flexível|
|**Consulta**|SQL padronizado|APIs específicas|
|**Escalabilidade**|Vertical (mais hardware)|Horizontal (mais nós)|
|**Consistência**|Forte (ACID)|Flexível (BASE)|
|**Performance**|Transações complexas|Leitura em grande escala|
|**Uso Ideal**|Sistemas transacionais|Big Data, conteúdo diverso|

## Propriedades ACID vs BASE

#### ACID - Bancos Estruturados

- ***"Atomicity":*** "Transações completas ou nada",
- ***"Consistency":*** "Estado válido sempre",
- ***"Isolation":*** "Transações independentes",
- ***"Durability"***: "Dados persistentes"

#### BASE - Bancos Não Estruturados

- ***"Basically Available"***: "Disponibilidade prioritária",
- ***"Soft state"***: "Estado pode mudar sem input",
- ***"Eventual consistency"***: "Consistência com o tempo"

# Modelos de Banco de Dados

## Outros Modelos Além do Relacional

1. **Orientado a Objetos**: Dados como objetos
2. **Hierárquico**: Estrutura em árvore
3. **Em Rede**: Relacionamentos complexos
4. **Documentos**: Dados em formato JSON/XML
5. **Grafos**: Foco em relacionamentos

## Exemplo de Modelo de Grafos

```cypher
// Cypher Query Language (Neo4j)
CREATE (alice:Person {name: 'Alice', age: 30})
CREATE (bob:Person {name: 'Bob', age: 25})
CREATE (alice)-[:KNOWS {since: 2020}]->(bob)
CREATE (alice)-[:WORKS_AT]->(company:Company {name: 'Tech Corp'})
```

# Aplicações Práticas

## Quando Usar Cada Tipo

**Banco Relacional (Estruturado):**

- Sistemas bancários    
- E-commerce com transações
- Aplicações que exigem consistência forte

**Banco Não Estruturado:**

- Redes sociais
- Catálogos de produtos
- Análise de grandes volumes de dados
- Conteúdo gerado por usuários

## Tendências Modernas

- **Bancos Poliglotas**: Uso múltiplos tipos conforme necessidade    
- **Híbridos**: Características de ambos os modelos
- **Cloud Native**: Otimizados para nuvem
- **Serverless**: Escala automática

# Considerações para Escolha

## Fatores de Decisão

1. **Natureza dos Dados**: Estruturados ou variáveis?
2. **Volume Esperado**: Pequeno/médio ou grande escala?
3. **Requisitos de Consistência**: Forte ou eventual?
4. **Padrão de Acesso**: Leitura ou escrita intensiva?
5. **Crescimento Esperado**: Previsível ou explosivo?

## Perguntas-Chave

- Qual a complexidade das consultas?
- Necessidade de transações ACID?
- Volume e velocidade de crescimento?
- Equipe familiarizada com a tecnologia?

---
## Conclusão

Os bancos de dados evoluíram significativamente desde os modelos relacionais tradicionais. A escolha entre estruturado e não estruturado depende das necessidades específicas do projeto. O cenário atual frequentemente envolve o uso combinado de diferentes tecnologias, aproveitando os pontos fortes de cada abordagem.

**"Não existe melhor ou pior, existe o mais adequado para cada contexto e necessidade."**
